Contexte du projet
Cette deuxième partie du mini-projet vise à mettre en place Flyway pour gérer les migrations de base de données (versionnement du schéma), mettre en place des DTO et appliquer une validation simple des entrées et dockeriser l’application et exécuter 2 conteneurs (conteneur MySQL, conteneur Spring Boot)

Migrations de base de données avec Flyway
Migration V1 — Création des tables

Créer le fichier : V1__init.sql

Il doit créer les tables :

client
vehicule
chauffeur
livraison
Contraintes minimales attendues :

Clés primaires sur chaque table
client.email unique
vehicule.matricule unique
Migration V2 — Évolution du schéma

Créer le fichier : V2__defaults_and_constraints.sql

Les clés étrangères
des valeurs par défaut (ex : statut, disponible)
au moins une contrainte de cohérence (ex : capacite > 0)
DTO + Validation
Règle : Les controllers ne doivent pas exposer les Entities directement.

Validation minimale attendue

Utiliser @Valid dans les controllers
Ajouter des validations de base sur les Request DTO :

@NotBlank (champs texte)
@NotNull (IDs / enums)
@Positive (capacité)
@Email (email)
Gestion des erreurs

Créer un @RestControllerAdvice pour retourner une réponse HTTP 400 claire lorsque la validation échoue.

Docker (2 conteneurs, sans Docker Compose)
Dockerfile (application spring boot)

Créer un Dockerfile permettant de :

builder le jar (ou copier le jar si déjà généré)
exécuter l’application Spring Boot
Lancer MySQL (conteneur 1)

Démarrer un conteneur MySQL via docker run avec :

nom du conteneur
mot de passe root / utilisateur / base
port exposé
network
Lancer l’application (conteneur 2)

Démarrer le conteneur Spring Boot via docker run avec :

port exposé (ex: 8080)
variables d’environnement pour se connecter à MySQL
network
Attendu :

Au démarrage de l’application, Flyway applique automatiquement les migrations.
Swagger est accessible et l’API fonctionne.
